# iOS device setup

This guide prepares a real iPhone for Edge Veda development and testing.

Use a physical device for performance checks. The iOS Simulator is useful for UI work, but it does not represent real Metal GPU inference performance.

## Supported setup

The current quickstart path focuses on iOS. Use this guide for:

- local Flutter development on macOS;
- a physical iPhone connected by USB or trusted Wi-Fi debugging;
- Xcode-managed signing;
- CocoaPods-managed native dependencies;
- release or profile builds for inference tests.

## Prerequisites

| Requirement | Recommended check |
| --- | --- |
| macOS | Use a Mac that can run a recent Xcode version. |
| Xcode | `xcode-select -p` |
| Flutter SDK | `flutter --version` |
| CocoaPods | `pod --version` |
| Apple ID | Required in Xcode Signing & Capabilities. |
| Physical iPhone | `flutter devices` |
| Developer Mode | iPhone Settings → Privacy & Security → Developer Mode. |

## 1. Verify Flutter and Xcode

Run:

```bash
flutter doctor
```

Fix Xcode or iOS toolchain warnings before debugging Edge Veda issues.

Then confirm Xcode command line tools:

```bash
xcode-select -p
```

If this points to the wrong Xcode installation, select the correct one:

```bash
sudo xcode-select -s /Applications/Xcode.app/Contents/Developer
```

## 2. Install CocoaPods

Check CocoaPods:

```bash
pod --version
```

If it is missing, install it using your preferred Ruby setup. One common option is:

```bash
sudo gem install cocoapods
```

After installation, run:

```bash
pod repo update
```

## 3. Create or open the Flutter app

Create a new app:

```bash
flutter create edge_veda_ios_test
cd edge_veda_ios_test
```

Or open an existing Flutter app.

Add Edge Veda:

```bash
flutter pub add edge_veda
flutter pub get
```

## 4. Set the iOS deployment target

Open `ios/Podfile` and set the minimum iOS version:

```ruby
platform :ios, '13.0'
```

The default setup does not require manual XCFramework download. The native framework is resolved during `pod install`.

## 5. Install pods

Run:

```bash
cd ios
pod install
cd ..
```

If pods fail, do not open `.xcodeproj`. Use the workspace generated by CocoaPods:

```bash
open ios/Runner.xcworkspace
```

## 6. Configure signing in Xcode

In Xcode:

1. Open `ios/Runner.xcworkspace`.
2. Select the **Runner** target.
3. Open **Signing & Capabilities**.
4. Enable **Automatically manage signing**.
5. Select your Apple developer team or personal team.
6. Confirm that the bundle identifier is unique.

A free Apple ID is enough for development builds. App Store distribution requires a paid developer account.

## 7. Prepare the iPhone

On the iPhone:

1. Connect the device to the Mac.
2. Unlock the phone.
3. Tap **Trust This Computer** if prompted.
4. Enable **Developer Mode**:
   - Settings → Privacy & Security → Developer Mode.
5. Restart the device if iOS requires it.
6. Keep the phone unlocked during the first deploy.

Verify that Flutter sees the device:

```bash
flutter devices
```

## 8. Run the app on device

For UI checks:

```bash
flutter run
```

For inference performance:

```bash
flutter run --release
```

For performance with DevTools support:

```bash
flutter run --profile
```

Use release or profile builds when measuring generation speed, first-token latency, model load time, or thermal behavior.

## Simulator limitations

The iOS Simulator can run the app UI, but it is not a reliable target for inference performance.

| Target | Use for | Do not use for |
| --- | --- | --- |
| iOS Simulator | UI layout, navigation, basic integration checks | Metal GPU performance, realistic token speed, memory pressure |
| Physical iPhone debug build | Development, logs, first setup check | Performance conclusions |
| Physical iPhone release build | Realistic inference behavior | Step-by-step debugging |
| Physical iPhone profile build | Performance testing with some tooling | Final production benchmark |

## First-run expectations

The first device run may take longer because:

- Flutter builds and signs the app;
- iOS installs and trusts the developer profile;
- the model downloads for the first time;
- the runtime loads model weights into memory.

A model can take tens of seconds to load on first initialization. Do not judge runtime speed until the app is running in release or profile mode.

## Useful commands

Clean Flutter build output:

```bash
flutter clean
flutter pub get
```

Reinstall pods:

```bash
cd ios
pod deintegrate
pod install
cd ..
```

Clear CocoaPods cache:

```bash
pod cache clean --all
```

Open the correct Xcode workspace:

```bash
open ios/Runner.xcworkspace
```

List devices:

```bash
flutter devices
```

Run in release mode:

```bash
flutter run --release
```

## Device checklist before reporting a bug

Before reporting an Edge Veda issue, capture:

- device model;
- iOS version;
- app build mode: debug, profile, or release;
- model name and size;
- `contextLength`;
- whether `useGpu` is enabled;
- whether the app ran on simulator or physical device;
- full Xcode or Flutter logs;
- steps to reproduce.

## Common setup problems

| Symptom | Cause | Fix |
| --- | --- | --- |
| Device does not appear in `flutter devices` | Device is locked, not trusted, or Developer Mode is off. | Unlock the phone, trust the Mac, enable Developer Mode. |
| Xcode signing error | No development team selected. | Select a team in Runner → Signing & Capabilities. |
| Build opens `.xcodeproj` instead of workspace | CocoaPods dependencies are not loaded. | Open `ios/Runner.xcworkspace`. |
| `No such module 'edge_veda'` | Pods were not installed or workspace was not used. | Run `pod install` and reopen the workspace. |
| Pod version conflict | Stale pod state or cache. | Run `pod deintegrate`, `pod install`, or clear pod cache. |
| Inference is very slow | Running debug mode or simulator. | Use a physical iPhone and `flutter run --release`. |
